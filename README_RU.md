# R-VDCF

**Reliable Event-Driven Violation-Driven Constraint Field** — экспериментальная
архитектура итеративного вывода на разреженном графе факторов. Она не
пересчитывает все связи на каждом шаге, а измеряет нарушения ограничений,
оценивает надёжность факторов и запускает обучаемые локальные коррекции только
для наиболее полезных факторов.

[Архитектура](ARCHITECTURE.md) · [Полные результаты](RESULTS.md) ·
[Исходный JSON](results/results.json)

## Что это такое

В текущем эксперименте R-VDCF — самостоятельная архитектура, а не добавочный
loss и не модуль внутри Transformer. Узлы хранят категориальное состояние,
факторы задают отношения между узлами, а единственный механизм обмена
информацией — выбранные планировщиком обучаемые коррекции.

На каждой итерации система:

1. измеряет нарушение каждого ограничения;
2. обновляет оценку доверия к фактору по его поведению во времени;
3. выбирает небольшой набор факторов по нарушению, доверию, возрасту и
   небольшой доле исследования;
4. применяет ограниченную по величине обучаемую коррекцию к связанным узлам;
5. обновляет затронутые состояния и повторяет процесс.

## Это замена attention или Transformer?

Пока нет. R-VDCF заменяет плотный итеративный вывод там, где уже существует
разреженный граф явных ограничений. Attention одновременно находит
зависящие от содержимого связи между токенами; R-VDCF сейчас получает связи
готовыми. Поэтому для LLM сначала нужно научиться строить полезные языковые
факторы без скрытого возврата к квадратичному attention.

Это направление может стать альтернативным вычислительным ядром для
структурированных задач, но текущий код не является готовой заменой слоя
Transformer.

## Что показал эксперимент

Модель обучалась только на графах из 64 узлов, затем без дообучения проверялась
на 64, 256 и 1 024 узлах. При 10% испорченных ограничений:

| Узлы | R-VDCF | Разреженный без reliability | Плотный режим |
|---:|---:|---:|---:|
| 64 | **0.8964** | 0.8192 | 0.9330 |
| 256 | **0.8856** | 0.7966 | 0.9185 |
| 1 024 | **0.8843** | 0.7870 | 0.9161 |

R-VDCF выполняла около **10.2% обучаемых обработок факторов** относительно
плотного режима и заметно превосходила разреженный режим без оценки
надёжности. При этом плотный режим сохранял более высокую точность на шумных
данных.

Перенос по размеру получился сильным: между 64 и 1 024 узлами точность R-VDCF
снизилась примерно на 1.21 процентного пункта при 10% шума и на 1.72 пункта
при 20% шума.

## Важное ограничение по сложности

Топология факторов разреженная: `E = O(N)`. Обучаемая коррекция обрабатывает
примерно `rho * E` факторов, где `rho ≈ 0.1`. Однако текущий прототип всё ещё
сканирует `O(E)` скалярных приоритетов ради global top-k. Поэтому он не имеет
квадратичной сложности, но пока и не даёт ускорения по реальному времени:
служебные операции gather/scatter, top-k и запуски GPU-ядер дороже сэкономленных
маленьких MLP на данном масштабе.

## Где это может быть полезно

- LDPC и другие коды исправления ошибок;
- SAT/CSP, раскраска графа, расписания, Sudoku и структурный repair;
- факторные графы, sensor fusion и вероятностный вывод;
- анализ программ, ограничения типов и распространение dataflow;
- любые системы с дешёвой проверкой локального нарушения и дорогой коррекцией.

R-VDCF менее полезна, если связи неизвестны заранее, почти все связи плотные
или требуется поиск по содержимому, как в attention.

## Запуск

Нужны Python 3.10+ и PyTorch 2.1+.

```bash
python -m pip install -r requirements.txt
python rvdcf_experiment.py --smoke --device cuda
python rvdcf_experiment.py --device cuda --amp
```

Результаты и checkpoint сохраняются в `rvdcf_runs/`. Параметры можно менять
через `--steps`, `--seed`, `--batch-size`, `--eval-nodes` и `--output-dir`.

## Что проверять дальше

1. Запустить 5–10 seed и публиковать среднее, стандартное отклонение и худший
   результат.
2. Заменить global top-k на блочный или bucket-планировщик и измерить реальное
   время на 4K–64K узлах.
3. Сделать исследование `O(k)`, не создающее случайный вектор размера `E`.
4. Сравнить на LDPC с BP, min-sum и neural BP при одинаковом бюджете.
5. Провести ablation для trust, aging, exploration и influence cap.

## Граница заявления о новизне

Факторные графы, residual scheduling, обучаемые сообщения, gating и exploration
имеют аналоги в прежних работах. Проверяемая идея — собрать вокруг
reliability-aware, violation-driven разреженного исполнения самостоятельную
архитектуру. До обзора литературы и реальных бенчмарков нельзя честно заявлять,
что каждый элемент новый или что это первая такая система.
